# ADR-005: Three-Layer Architecture — Foundry, Executor, Monitor

**Date:** 2026-04-19  
**Status:** Accepted  
**Authors:** Dave + Pecz

## Context

Foundry V1 (Strategy Generation + Backtest + Walk-Forward + Gates) is approaching production readiness with the ADX+EMA regime-filtered MACD Momentum strategy achieving 50% pass rate across 8 assets. As we prepare for Paper/Shadow Trading, the question arises: what happens *after* a strategy passes the Foundry gates?

Currently, Foundry handles everything from generation to evaluation. But live trading requires:
- Real-time order execution and position management
- Continuous monitoring with kill-switches
- Post-trade review and strategy retirement

These are fundamentally different responsibilities that should not be mixed in one system.

## Decision

We adopt a **three-layer architecture** with clear separation of concerns:

```
┌─────────────────────────────────────────────────────┐
│  LAYER 1: FOUNDRY (Pre-Trade)                       │
│  Strategy Generation → Backtest → Walk-Forward       │
│  → Gates → Paper/Shadow Trading Validation           │
│  Output: Approved strategy specs                     │
│  Technology: Python, Polars, LLM Generator           │
├─────────────────────────────────────────────────────┤
│  LAYER 2: EXECUTOR (During Trade)                   │
│  Order Management → Position Tracking → Exits        │
│  Kill-Switches → Risk Limits → Position Sizing       │
│  Input: Approved strategy specs from Foundry         │
│  Technology: Hyperliquid SDK, deterministic rules     │
├─────────────────────────────────────────────────────┤
│  LAYER 3: MONITOR (Post-Trade / Continuous)          │
│  PnL Tracking → Regime Detection → Strategy Review   │
│  Retirement Signals → Performance Degradation Alerts  │
│  Technology: Dashboard, alerts, periodic reviews      │
└─────────────────────────────────────────────────────┘
```

### Key Principles

1. **Foundry is NOT the live monitor.** Foundry generates, validates, and approves strategies. Once a strategy is live, Foundry's job is done.

2. **No second free-ranging AI as judge.** Gates are deterministic, not probabilistic. The Executor follows rules, not vibes.

3. **Deterministic execution.** The Executor uses fixed rules for entries, exits, position sizing, and risk limits. No LLM calls during trading.

4. **Kill-Switches are non-negotiable.** Daily-loss limits, max drawdown, max open positions, and time-based circuit breakers are hardcoded in the Executor, not configurable by the strategy.

5. **Strategy retirement is data-driven.** A strategy is retired when its rolling metrics degrade below Foundry gate thresholds — not because an AI "feels" it should be retired.

### Layer Interfaces

```
Foundry → Executor: Approved strategy spec (JSON)
  - Entry/exit conditions
  - Position sizing rules
  - Risk parameters
  - Regime filters

Executor → Monitor: Trade events (JSON)
  - Entries, exits, PnL
  - Kill-switch activations
  - Position state changes

Monitor → Foundry: Retirement signals (JSON)
  - Performance degradation metrics
  - Regime change detection
  - Trigger for re-validation
```

## V1 Scope (Current Priority)

**Foundry (Layer 1):** Continue current work
- ADX+EMA regime filter validation → ≥60% pass rate
- Paper/Shadow trading on Hyperliquid testnet
- Gate thresholds finalized before live

**Executor (Layer 2):** Build after Foundry validates
- Hyperliquid SDK integration
- Deterministic order execution
- Hardcoded risk limits (2% per trade, 5% daily loss, 1 position max)
- Kill-switches wired to Discord alerts

**Monitor (Layer 3):** Build after Executor is running
- Rolling PnL dashboard
- Regime change detection (ADX, EMA crossovers)
- Strategy health scoring
- Retirement criteria (same gates as Foundry, applied to live data)

## Consequences

- **Positive:** Clear separation of concerns. Each layer can be developed, tested, and deployed independently.
- **Positive:** Foundry remains pure (generation + validation). No scope creep into live trading.
- **Positive:** Deterministic execution means reproducible behavior. No "AI decided differently" scenarios.
- **Negative:** Three systems to maintain instead of one. Each needs its own deployment and monitoring.
- **Negative:** Interface contracts between layers must be stable and versioned.

## Out of Scope (Explicitly Deferred)

- Multi-strategy portfolio orchestration (V2+)
- LLM-driven strategy adaptation during live trading (never — deterministic only)
- Second AI as judge or override (never — deterministic gates only)

## Optional: News / Macro Risk Layer (V2+)

**Status:** Deferred, not rejected. Could add value as a **risk modifier**, not a decision-maker.

### Distinction: Signal vs. Decision

| Rolle | Beispiel | Erlaubt? |
|-------|----------|----------|
| **KI als Richter** | "Soll ich traden? KI entscheidet." | ❌ NEIN |
| **KI als Signalgeber** | "Makro-Risiko-Score: 85/100 → halbiere Position" | ✅ JA |

### Wie ein Makro-Risk-Layer funktionieren würde:

```
News/Macro Risk (KI-basiert, optional)
    ↓
    Liefert: Risk Score 0-100
    ↓
Executor (deterministisch)
    Risk Score > 80 → Position Size × 0.5
    Risk Score > 95 → PAUSE (keine neuen Trades)
    Risk Score < 80 → Normale Position Size
```

### Design-Prinzipien

1. **KI bewertet, entscheidet nicht.** Der Risk-Score ist ein Input für deterministische Regeln.
2. **Default ist handelsfrei.** Wenn der Risk-Layer offline ist, tradet der Executor normal weiter.
3. **Nur Downsizing, nie Upsizing.** KI kann Risiken erkennen und Positionen verkleinern – niemals vergrößern.
4. **Transparent.** Jeder Risk-Score wird geloggt und ist nachvollziehbar.
5. **Kein Override.** KI kann keine Kill-Switches umgehen oder Gates überspringen.

### Beispiele für Makro-Risiken

- Fed-Rate-Entscheidung in 2h → Score 75 → Position halbiert
- Kriegsausbruch / geopolitischer Schock → Score 95 → Trading pausiert
- Crypto-spezifisch: Exchange-Hack, regulatorischer Einschlag → Score 85 → Position halbiert
- Kein besonderes Ereignis → Score 0 → Normales Trading

### V2+ Roadmap

1. **V2a:** Manueller Risk-Score via Dashboard (Dave setzt Score 0-100)
2. **V2b:** Semi-automatisch: KI liest Kalender (FOMC, CPI, etc.) → Score-Vorschlag → Dave bestätigt
3. **V2c:** Vollautomatisch: LLM analysiert News-Feeds → Score → Deterministische Aktion

Stufe 2c ist die späteste, weil sie die höchste Komplexität hat.
