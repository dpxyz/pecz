# Block 5.3.1: Health Endpoint Design

**Status:** Design Phase  
**Pattern:** Kubernetes/Cloud-Native Health Checks  
**Priority:** Function over Form  

---

## Design-Prinzipien

1. **Klare Signale zuerst** - Jeder Endpoint hat EINE Aufgabe
2. **Keine Überladung** - Nur kritische Metriken
3. **HTTP-Semantik korrekt** - 200 = healthy, 503 = unhealthy
4. **Schnell** - Keine langsamen Checks in /live
5. **Konsistent** - Gleiche JSON-Struktur über alle Endpoints

---

## Endpoints

### 1. /health/live (Liveness)

**Zweck:** Ist der Prozess am Leben?

**Checks:**
- Prozess läuft (PID existiert)
- Kein Deadlock (letzter Heartbeat < 60s)
- Memory nicht unendlich gewachsen

**Response (200):**
```json
{
  "status": "UP",
  "component": "liveness",
  "timestamp": "2026-04-01T11:00:00.000Z",
  "uptime_seconds": 86400
}
```

**Response (503):**
```json
{
  "status": "DOWN",
  "component": "liveness",
  "timestamp": "2026-04-01T11:00:00.000Z",
  "reason": "Heartbeat_stale",
  "last_heartbeat": "2026-04-01T10:58:00.000Z",
  "threshold_seconds": 60
}
```

**Wann 503:**
- Kein Heartbeat seit >60s
- OOM (Memory >95%)

**Performance:** < 10ms

---

### 2. /health/ready (Readiness)

**Zweck:** Ist der Service bereit, Anfragen zu bearbeiten?

**Checks:**
- Event Store erreichbar (DB date modified < 5min)
- Circuit Breaker nicht OFFEN
- Kein Startup-Mode aktiv
- State Projection aktuell

**Response (200):**
```json
{
  "status": "UP",
  "component": "readiness",
  "timestamp": "2026-04-01T11:00:00.000Z",
  "checks": {
    "event_store": "OK",
    "circuit_breaker": "CLOSED",
    "state_projection": "CURRENT"
  },
  "ready_since": "2026-04-01T09:57:07.000Z"
}
```

**Response (503):**
```json
{
  "status": "DOWN",
  "component": "readiness",
  "timestamp": "2026-04-01T11:00:00.000Z",
  "checks": {
    "event_store": "STALE (last update 600s ago)",
    "circuit_breaker": "OPEN",
    "state_projection": "REBUILDING"
  },
  "reason": "circuit_breaker_open"
}
```

**Wann 503:**
- Event Store nicht erreichbar
- Circuit Breaker OPEN
- State wird gerade rebuilt

**Performance:** < 50ms (DB check)

---

### 3. /health/startup

**Zweck:** Ist der Service gerade am Starten?

**Checks:**
- Startup-Phase aktiv (erste 2 Minuten ODER explizites Flag)

**Response (200):**
```json
{
  "status": "UP",
  "component": "startup",
  "timestamp": "2026-04-01T11:00:00.000Z",
  "startup_phase": "COMPLETE",
  "startup_duration_seconds": 45
}
```

**Response (503 - während startup):**
```json
{
  "status": "DOWN",
  "component": "startup",
  "timestamp": "2026-04-01T11:00:00.000Z",
  "startup_phase": "IN_PROGRESS",
  "progress_percent": 75,
  "message": "Rebuilding state from event store..."
}
```

**Wann 503:**
- Erste 120 Sekunden nach Start
- ODER explizites "starting" Flag gesetzt
- State Rebuild läuft

**Performance:** < 1ms (nur Memory-Check)

---

### 4. /health (Aggregiert - Optional)

**Zweck:** Übersicht über alle Health-Checks

**Checks:** Kombiniert live + ready

**Response (200 - alles OK):**
```json
{
  "status": "UP",
  "timestamp": "2026-04-01T11:00:00.000Z",
  "uptime_seconds": 86400,
  "components": {
    "liveness": "UP",
    "readiness": "UP",
    "startup": "COMPLETE"
  },
  "summary": "All systems operational"
}
```

**Response (503):**
```json
{
  "status": "DOWN",
  "timestamp": "2026-04-01T11:00:00.000Z",
  "uptime_seconds": 86400,
  "components": {
    "liveness": "UP",
    "readiness": "DOWN",
    "startup": "COMPLETE"
  },
  "reason": "circuit_breaker_open"
}
```

---

### 5. /health/deep (Optional/Expensive)

**Zweck:** Tiefergehende Checks (nicht für Probes)

**Checks:**
- Event Store Integrity (SELECT COUNT)
- Alle Services können erreicht werden
- Memory Leak Check (Growth über 1h)
- Disk Space verfügbar

**Response:** Ähnlich wie /health, aber mit details Array

**Performance:** 100-500ms (akzeptabel, da nicht für K8s probes)

---

## HTTP Status Codes

| Endpoint | 200 OK | 503 Service Unavailable |
|----------|--------|-------------------------|
| /live | Prozess lebt, Heartbeat frisch | Prozess tot oder stale |
| /ready | Alle Checks pass | Irgendein Check fail |
| /startup | Startup complete | Startup läuft noch |
| /health | live=UP UND ready=UP | Einer der beiden DOWN |
| /deep | Alle Checks pass | Irgendein Check fail |

**WICHTIG:** 
- Keine 500er für Health Checks (nur 200 oder 503)
- 503 = Kubernetes restartet Container nicht direkt

---

## JSON Schema (alle Endpoints)

```typescript
interface HealthResponse {
  status: "UP" | "DOWN";
  component: "liveness" | "readiness" | "startup" | "aggregated";
  timestamp: string; // ISO 8601
  // Optional je Endpoint:
  uptime_seconds?: number;
  checks?: Record<string, string>;
  reason?: string;
  components?: Record<string, string>;
}
```

## Implementierungshinweise

### Express.js Routes

```javascript
app.get('/health/live', (req, res) => {
  const healthy = checkLiveness();
  res.status(healthy ? 200 : 503).json({
    status: healthy ? 'UP' : 'DOWN',
    component: 'liveness',
    timestamp: new Date().toISOString()
  });
});
```

### Caching

- /live: Kein Cache (sehr schnell)
- /ready: Max 5s Cache (event_store modified check)
- /deep: Max 60s Cache (teuer)

### Logging

- Health Check Zugriffe NICHT im normalen Log (zu viel noise)
- Separate Log-Level: health-checks.log
- Nur 503 Fehler loggen
